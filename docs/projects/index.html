Active Directory Migration Plan

Identity & Access • Windows Server • GPO

Identity & Access

Executive Summary

Designed a migration plan to move from a flat, legacy Active Directory structure to a role-based OU and group model. The plan focuses on improving manageability, security, and clarity for future administration.

Business Problem

The existing domain used a single OU with inconsistent group naming and no clear separation between departments or administrative boundaries. This made it difficult to apply targeted Group Policy, audit access, or delegate administration.

Technical Solution

I created a new OU structure aligned to departments and roles, standardized group naming conventions, and defined GPO baselines for workstations and servers. A staged migration plan was documented to move users, computers, and policies with minimal disruption.

Architecture Overview

Root Domain: lab.local
├── OU=Corp
│   ├── OU=Users
│   │   ├── OU=HR
│   │   ├── OU=IT
│   │   └── OU=Finance
│   ├── OU=Computers
│   │   ├── OU=Workstations
│   │   └── OU=Servers
│   └── OU=Groups
│       ├── OU=Security
│       └── OU=Distribution
└── OU=Service
    ├── OU=ServiceAccounts
    └── OU=Admin

Outcome

  • Clear separation of users, computers, and groups by department and function
  • GPOs targeted to specific OUs with reduced policy conflicts
  • Improved ability to delegate administration to department IT contacts

Lessons Learned

  • Planning OU design around business structure simplifies long-term management
  • Standardized group naming reduces confusion during troubleshooting
  • Documented migration steps help avoid accidental policy changes

Technologies Used

  • Windows Server 2019
  • Active Directory Domain Services
  • Group Policy Management
  • PowerShell (ActiveDirectory module)

Key Artifacts

  • Lab architecture overview
  • User provisioning script
  • Change record

New-ADOrganizationalUnit -Name "Groups" -Path "OU=Corp,DC=lab,DC=local"

Microsoft 365 Personal Cloud Lab

Microsoft 365 • Entra ID • Intune

Cloud & Identity

Executive Summary

Built a Microsoft 365 tenant that mirrors a small-business environment, including Entra ID, Exchange Online, SharePoint, Teams, and Intune integration. The lab is used to practice administration, automation, and troubleshooting.

Business Problem

Many entry-level roles expect familiarity with Microsoft 365 administration, but production access is limited. I needed a safe environment to practice tenant configuration, identity management, and endpoint integration.

Technical Solution

I provisioned a Microsoft 365 tenant, configured Entra ID with security groups and roles, set up Exchange Online mailboxes, created SharePoint team sites, and integrated Intune for device management. I also created automation scripts using Microsoft Graph.

Architecture Overview

[Users] --> [Entra ID Tenant]
   |             |
   |             +--> [Microsoft 365 Apps]
   |                    - Exchange Online
   |                    - SharePoint Online
   |                    - Teams
   |
   +--> [Intune]
          |
          +--> [Windows 11 Devices]

Outcome

  • Hands-on experience with core Microsoft 365 administration tasks
  • Working integration between identity, collaboration, and endpoint management
  • Reusable scripts for user and group management via Microsoft Graph

Lessons Learned

  • Licensing and group-based assignment simplify user onboarding
  • Conditional Access policies should be tested with pilot groups first
  • Documenting tenant configuration helps when revisiting settings later

Technologies Used

  • Microsoft 365
  • Entra ID
  • Exchange Online
  • SharePoint & Teams
  • Intune

Key Artifacts

  • Tenant architecture documentation

Intune Baseline Deployment

Endpoint Management • Security Baselines

Endpoint Management

Executive Summary

Implemented a secure workstation baseline for Windows 11 devices using Intune compliance and configuration policies, focusing on encryption, password policies, and endpoint protection.

Business Problem

Devices in the lab environment had inconsistent security settings, making it difficult to ensure that all endpoints met a minimum security standard.

Technical Solution

I created Intune compliance policies for encryption and OS version, configuration profiles for password and lock screen policies, and endpoint security baselines. Deployment was validated using device reports and remediation logs.

Architecture Overview

[Intune]
  ├── Compliance Policies
  │     - Require BitLocker
  │     - Minimum OS version
  ├── Configuration Profiles
  │     - Password & lock screen
  │     - Defender settings
  └── Security Baselines
        - MDM Security Baseline

Outcome

  • All enrolled Windows 11 devices reported compliant with baseline policies
  • Documented deployment steps and troubleshooting notes
  • Improved understanding of Intune reporting and policy conflicts

Lessons Learned

  • Testing baselines on a pilot group prevents widespread configuration issues
  • Compliance policies should be paired with clear remediation guidance
  • Exporting JSON configurations helps track changes over time

Technologies Used

  • Intune
  • Windows 11
  • Endpoint security baselines

Incident Root Cause Analysis

Troubleshooting • DNS • Authentication

Troubleshooting

Executive Summary

Investigated intermittent authentication failures affecting domain-joined clients and cloud sign-ins. The root cause was traced to a misconfigured DNS forwarder on a branch domain controller.

Business Problem

Users reported occasional sign-in failures and delays when accessing resources. The issue was intermittent, making it difficult to reproduce and diagnose quickly.

Technical Solution

I collected Event Viewer logs, DNS logs, and client-side error messages, then correlated timestamps to identify patterns. Testing with nslookup and Test-ComputerSecureChannel revealed inconsistent name resolution. Updating DNS forwarders and clearing caches resolved the issue.

Architecture Overview

[Clients] --> [Branch DC / DNS] --> [Internet DNS]
                 |
                 +--> [Hub DC / DNS]

Outcome

  • Authentication failures stopped after DNS configuration was corrected
  • Documented incident timeline and corrective actions
  • Added DNS configuration checks to regular maintenance tasks

Lessons Learned

  • DNS misconfigurations can present as authentication issues
  • Correlating logs across systems is critical for RCA
  • Documented change management helps prevent regressions

Technologies Used

  • Windows Server DNS
  • Active Directory
  • Event Viewer
  • PowerShell

Key Artifacts

PowerShell User Provisioning Automation

Automation • Identity • Microsoft Graph

Automation

Executive Summary

Developed a PowerShell script that automates new hire provisioning across Active Directory and Entra ID, including group membership and license assignment.

Business Problem

Manual user creation is time-consuming and error-prone, especially when multiple systems and groups must be updated consistently.

Technical Solution

I created a script that reads a JSON template for new hires, creates accounts in AD, syncs to Entra ID, assigns licenses via Microsoft Graph, and logs all actions. The script supports dry-run mode for testing.

Architecture Overview

[New Hire JSON] --> [PowerShell Script]
      |                    |
      |                    +--> [Active Directory]
      |                    +--> [Entra ID / Microsoft 365]
      |                    +--> [Log File]

Outcome

  • Reduced manual steps for new hire onboarding
  • Consistent group membership and license assignment
  • Clear logs for auditing and troubleshooting

Lessons Learned

  • Dry-run mode is essential for safe testing
  • JSON templates make it easy to adjust onboarding rules
  • Logging should capture both success and failure details

Technologies Used

  • PowerShell
  • ActiveDirectory module
  • Microsoft Graph PowerShell SDK

Key Artifacts

  • RCA and provisioning artifacts
  • Automation skill overview

Script Excerpt

param(
  [string]$InputFile = ".\new-hires.json",
  [switch]$WhatIf
)

$users = Get-Content $InputFile | ConvertFrom-Json
foreach ($user in $users) {
  Write-Host "Creating user $($user.DisplayName)..." -ForegroundColor Cyan
  if (-not $WhatIf) {
    New-ADUser -Name $user.DisplayName -SamAccountName $user.SamAccountName `
      -UserPrincipalName $user.UserPrincipalName -GivenName $user.GivenName `
      -Surname $user.Surname -Path $user.OU -Enabled $true
  }
}
